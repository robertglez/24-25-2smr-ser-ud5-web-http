03| Etapas de una transacción HTTP

Con la intención de conocer con más profundidad el protocolo HTTP, podemos evaluar en qué consiste una transacción HTTP.

Cada vez que un cliente realiza una petición a un servidor, se ejecutan una serie de acciones:

  1. Un usuario accede a una dirección de Internet (URL) seleccionando un enlace de un documento HTML o introduciéndola directamente en la barra de navegación de un navegador web desde la perspectiva del cliente web. El cliente web descodifica la dirección de Internet (URL) separando sus diferentes partes. Así se identifican el protocolo de acceso, el nodo expresado con el nombre de dominio o su dirección IP, el posible puerto opcional (el valor por defecto es el 80) y el objeto del servidor requerido.
  2. Se abre una conexión TCP/IP con el servidor llamando al puerto TCP correspondiente. Se realiza la petición. En consecuencia, se envían la orden necesaria (GET, POST, HEAD, etc.), la dirección del objeto requerido (el contenido de la dirección de Internet del servidor), la versión del protocolo HTTP utilizada (en la mayoría de los casos es HTTP/1.0) y un conjunto variable de información que incluye datos sobre las capacidades del navegador web, datos opcionales para el servidor, etc.
  3. El servidor localiza el recurso solicitado y devuelve la respuesta al cliente.
  4. Esta respuesta consiste en un código de estado y el tipo de dato con extensiones multipropósito de correo de Internet (MIME, Multipurpose Internet Mail Extension) de la información de respuesta, seguido de la propia información.
  5. El cliente formatea y muestra el recurso recibido.
  6. Se cierra la conexión TCP.

Este proceso se repite en cada acceso que se realice al servidor HTTP. Por ejemplo, si se solicita un documento HTML que contiene cuatro imágenes, el proceso de transacción mostrado anteriormente se repite cinco veces, es decir, una para el documento HTML y cuatro para las imágenes.

Etapas en la transacción HTTP

Extensión MIME

El protocolo HTML fue diseñado para transportar por la red archivos en formato ASCII. Con el avance de las tecnologías y la inclusión de diferentes tipos de archivos en un formato distinto al ASCII (imágenes, vídeos, sonidos, etc.) en las aplicaciones para Internet, surgió la necesidad de transformar estos formatos a ASCII para su correcta recepción en el navegador web.

En 1992, se crearon los tipos MIME, que eran especificaciones para dar formato a mensajes en un formato distinto al ASCII para que pudieran ser enviados e interpretados en Internet. Más adelante, se aplicaron también a los documentos web.

Nomenclatura de tipos

MIME asigna un nombre a cada tipo de datos. Los nombres siguen el siguiente formato:

tipo / subtipo (tanto tipo como subtipo son cadenas de caracteres)

El tipo define la categoría general de los datos y el subtipo define un tipo más específico dentro de esta categoría. Los tipos pueden tener los siguientes valores:

  • text: Indica que el contenido es texto plano. Ejemplos de subtipos: html, xml
  • multipart: Indica que contiene múltiples partes de datos independientes. Ejemplos de subtipos: form-data, mixed
  • message: Para encapsular un mensaje existente. Por ejemplo, cuando queremos responder a un mensaje de correo incorporando el mensaje original. Ejemplos de subtipos: partial, RFC822
  • image: Indica que es una imagen. Ejemplos de subtipos: png, gif
  • audio: Indica que es un archivo de audio. Ejemplos de subtipos: mpeg, wav
  • video: Indica que es un archivo de vídeo. Ejemplos de subtipos: mp4, 32kadpcm
  • application: Indica que se trata de datos de aplicación que pueden ser binarios. Ejemplos de subtipos: json, pdf
